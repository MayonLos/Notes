> **一句话定义**：渐进式加载是一种设计模式，将信息按使用频率和优先级分层，只在实际需要时才加载更详细的内容，从而在有限资源（如 LLM 上下文窗口）下最大化效率。

## 详细说明

### 问题背景

在 AI Agent 系统中，上下文窗口是昂贵且有限的资源。如果将所有可用工具、指令、参考资料一次性全部加载，会出现两个问题：

1. **上下文浪费**：大部分内容与当前任务无关，占用宝贵 token
2. **注意力稀释**：过多无关信息会降低 Agent 的任务聚焦能力

### 三层架构

渐进式加载在 Agent Skills 中通过三层实现：

```
┌─────────────────────────────────────┐
│  第 1 层：元数据索引                 │
│  ~100 tokens / skill                │
│  启动时全部加载                      │
│  内容：name + description           │
│  目的：让 Agent 知道"有哪些能力可用"  │
├─────────────────────────────────────┤
│  第 2 层：完整指令                   │
│  < 5000 tokens（推荐）              │
│  激活 skill 时加载                   │
│  内容：SKILL.md 正文                │
│  目的：告诉 Agent "具体怎么做"       │
├─────────────────────────────────────┤
│  第 3 层：按需资源                   │
│  无固定上限                          │
│  执行中按需加载                      │
│  内容：scripts/ references/ assets/ │
│  目的：提供"执行时需要的具体材料"    │
└─────────────────────────────────────┘
```

### 设计优势

| 优势 | 说明 |
|:---|:---|
| **上下文效率** | 50 个 skill 启动只需 ~5000 tokens，而非 25 万 |
| **注意力聚焦** | Agent 只看到与当前任务相关的内容 |
| **可扩展性** | 添加新 skill 不影响已有 skill 的加载成本 |
| **模块化** | 每个 skill 独立维护，互不干扰 |

### 与其他模式的对比

- **懒加载（Lazy Loading）**：渐进式加载是懒加载在 AI 上下文管理中的特化应用
- **按需加载（On-Demand Loading）**：第 3 层即典型的按需加载
- **索引优先（Index-First）**：第 1 层的设计思想，类似数据库索引——先查索引，再读数据

## 在其他领域的应用

虽然此概念在 Agent Skills 中被明确提出，但其思想广泛适用：

- **Web 性能优化**：图片懒加载、代码分割（code splitting）
- **数据库**：索引 + 按需查询数据页
- **文档系统**：目录 → 章节摘要 → 全文
- **UI 设计**：渐进式表单、折叠面板
